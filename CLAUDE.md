# CLAUDE.md — Production LLMOps: AI Application Lifecycle

## Project Identity
This project generates the **Production LLMOps: AI Application Lifecycle** course — a 32-module program across 8 tracks, teaching ML engineers, SREs, and engineering managers how to evaluate, deploy, monitor, upgrade, and govern LLM applications in production. All modules are anchored in the **Prior Authorization (PA) Determination Pipeline** domain — a health-plan LLM application that reads incoming auth requests (member, requested CPT/HCPCS, diagnosis ICD-10, clinical notes), matches them against payer medical-policy criteria, and recommends **approve / deny / pend** with nurse and medical-director review. The course uses a **dark theme**.

Published at **agenticai.varasrinivas.com** as part of the Agentic AI five-pillar curriculum:
- **Context Engineering** → science of context design
- **Building AI Agents with Claude** → applied practice
- **AI Platform Engineering** → infrastructure layer
- **Production LLMOps** (this course) → operational lifecycle
- **AI-SDLC Series** → enterprise delivery process

## Architecture

### Single-File HTML Player
- One `course/index.html` — `MODS` array + `renderVisual()` switch + `TRACK_META`
- Module navigation, progress tracking, keyboard shortcuts self-contained

### Build System (Generator)
The full course is generated by Python scripts so content stays maintainable:
- `scripts/modules_t1_4.py` — modules M00–M15 + shared SVG helper library
- `scripts/modules_t5_8.py` — modules M16–M31
- `scripts/generate_course.py` — dark-theme HTML shell + JS engine; assembles `course/index.html`

Regenerate the whole course with: `python scripts/generate_course.py`
Edit a module by changing its dict in the relevant `modules_*.py` file, then regenerate.

### Content Rules (12-Rule Depth System)
1. Every concept gets an everyday analogy FIRST, then technical explanation
2. Analogies from: restaurants, hospitals, factories, airlines, fire departments, pharmacies
3. No concept without a concrete Prior Authorization (PA) domain example
4. Code focuses on operational tooling (eval scripts, monitoring configs, CI/CD pipelines, runbooks)
5. Every module: 2 labs — "Understand It" (observe/audit) + "Build It with AI" (construct with Claude)
6. SVG diagrams in `renderVisual()`, no external images
7. 4-6 key topics, one analogy, cross-links where applicable
8. Quiz questions test WHEN to apply a practice and WHY, not definitions
9. Progressive complexity: M+0 = "what is this", M+3 = "production edge cases"
10. Cross-links: `[CE MXX]`, `[Agent MXX]`, `[Platform MXX]`, `[SDLC Track X]`
11. Anti-patterns with real production failure stories and consequences
12. Every module ends with an "LLMOps Takeaway" — one sentence

## Editing Modules
Preferred path: edit the module dict in `scripts/modules_t1_4.py` / `scripts/modules_t5_8.py`, then run `python scripts/generate_course.py`. This keeps `MODS`, `TRACK_META`, and the `renderVisual()` switch in sync and handles all escaping.

If you must inject directly into a hand-edited `course/index.html` instead of regenerating, use Python (never rewrite the full file). Note the current generated markers:
```python
import json
with open("course/index.html", "r", encoding="utf-8") as f: src = f.read()
marker = "];\nconst TRACK_META"                       # no blank line in generated output
src = src.replace(marker, f",\n{json.dumps(module_obj, indent=2)}\n{marker}")
visual_marker = "    default: return '';"             # 4-space indent in generated output
# SVG cases are emitted as JSON-escaped strings, not backtick templates:
src = src.replace(visual_marker, f'    case "{key}": return {json.dumps(svg)};\n{visual_marker}')
with open("course/index.html", "w", encoding="utf-8") as f: f.write(src)
```

2. All required fields present
3. Analogy BEFORE technical explanation

6. 4-6 key topics

8. Anti-patterns with production consequences
9. Cross-links tagged where applicable

11. SVG diagram in renderVisual()
12. `</script>` escaped as `<\/script>`
13. No trailing commas in MODS array
14. Progressive complexity for track position

18. Valid section types
19. Track color matches TRACK_META
20. Module ID follows MXX format
